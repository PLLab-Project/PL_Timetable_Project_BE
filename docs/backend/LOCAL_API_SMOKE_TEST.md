# 로컬 API 실사용 스모크 테스트

## 실행 기준

- 실행일: 2026-07-24
- 기준 브랜치: `fix/spa-csrf-smoke`
- 데이터베이스: PostgreSQL 18.4
- Flyway 최신 버전: `2026.07.23.006`
- 테스트 서버: `http://localhost:18082`
  - 로컬의 `8080` 포트가 이미 사용 중이어서 테스트에서만 `18082`를 사용했다.
- 테스트 전에 전체 DB 백업을 생성하고 마이그레이션 및 정규화 적재를 다시 실행했다.

- OTP 원문, 세션 쿠키, CSRF 토큰은 이 문서에 기록하지 않는다.

## 최종 테스트 결과

### 공통·보안

| 케이스 | 기대 상태 | 실제 상태 | 결과 |
|---|---:|---:|---|
| 서버 생존 확인 | 200 | 200 | 통과 |
| 로그인 없이 보호 API 접근 | 401 | 401 | 통과 |
| CSRF 토큰 없이 변경 요청 | 403 | 403 | 통과 |
| SPA 쿠키 토큰을 헤더에 포함한 변경 요청 | 2xx | 2xx | 통과 |
| 필수 쿼리 누락 | 400 | 400 | 통과 |
| 없는 학사 리소스 조회 | 404 | 404 | 통과 |

CSRF는 최초 테스트에서 서버가 발급한 쿠키 토큰을 그대로 전송해도 모든 변경 요청이
`403`으로 차단되는 결함이 있었다. Spring Security의 SPA CSRF 설정을 적용하고 회귀
테스트를 추가한 뒤 실제 요청이 통과하는 것을 확인했다.

|---|---:|---|

| 잘못된 학년 수정 | 400 | 검증 오류 |
| 개인정보 동의 저장·조회 | 200 | 동의 버전과 시각 반환 |

### 시간표

| 케이스 | 실제 상태 | 결과 |
|---|---:|---|
| 존재하지 않는 분반으로 생성 | 400 | 통과 |
| 시간표 생성·목록·단건 조회 | 201/200 | 통과 |
| 이름 변경 | 200 | 통과 |
| 분반 구성 전체 교체 | 200 | 통과 |
| 분반 추가 | 201 | 통과 |
| 분반 삭제 | 200 | 통과 |
| 같은 분반 중복 추가 | 409 | `SECTION_CONFLICT` |
| 시간표 삭제 | 204 | 통과 |
| 삭제한 시간표 조회 | 404 | 통과 |

최초 테스트에서는 분반 추가·교체 직후 응답의 새 분반 `id`가 `null`이었다. DB flush
시점을 보완하고 회귀 테스트를 추가한 뒤, 추가·교체 응답에서 즉시 유효한 `id`가
반환되고 그 ID로 삭제할 수 있음을 실제 요청으로 다시 확인했다.

### 자동 편성

| 케이스 | 실제 상태 | 결과 |
|---|---:|---|
| 작업 생성 | 201 | `PENDING` 반환 |
| 비동기 처리 완료 조회 | 200 | `SUCCESS`와 결과 반환 |
| 완료된 작업 취소 | 409 | `OPTIMIZATION_ALREADY_FINISHED` |

자동 편성 결과 응답 예:

```json
{
  "status": "SUCCESS",
  "results": [
    {
      "rank": 1,
      "sections": [
        {
          "courseCode": "855121",
          "sectionCode": "02",
          "dayOfWeek": "THURSDAY",
          "startTime": "14:30:00",
          "endTime": "17:30:00"
        }
      ],
      "attendanceDays": 1,
      "totalCredits": 2.0,
      "totalFreeMinutes": 0,
      "score": 135.0
    }
  ]
}
```

최초 테스트에서는 완료된 결과를 조회할 때 Hibernate 지연 로딩 예외로 `500`이
발생했다. 조회와 DTO 변환을 동일한 읽기 트랜잭션 안에서 수행하도록 수정한 뒤
회귀 테스트와 실제 `200/SUCCESS` 응답을 확인했다.

판정 응답에는 총학점·전공·교양 부족분, 영역 부족분, 필수과목, 추천 강의,

단, 현재 사용자 수정 API에는 `admissionYear`, `studentType`, `programPath` 입력이
없다. 이번 테스트에서는 전용 테스트 계정의 해당 값만 DB fixture로 넣어 판정했다.
따라서 프론트 온보딩만으로 졸업 판정까지 연결하려면 프로필 입력 API 보완이 필요하다.

## 현재 API로 실사용 테스트할 수 없는 항목

다음 항목은 대응하는 구현 또는 엔드포인트가 없어 이번 스모크 테스트 대상이 아니다.

- 소셜 로그인과 공급자 콜백·계정 연결
- 시간표 즐겨찾기
- 시간표 복사
- 이전 학기 시간표 조회
- 공유 링크 생성·조회
- 다른 분반으로 교체하는 전용 API
- 충돌 없는 대체 분반 추천 API
- OCR 업로드·인식·수정·원본 삭제
